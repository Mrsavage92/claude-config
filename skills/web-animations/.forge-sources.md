### Source 1: Motion.dev — useReducedMotion Hook
- **URL**: https://motion.dev/docs/react-use-reduced-motion
- **Pattern**: The `useReducedMotion()` hook returns a reactive boolean so components can swap motion-heavy transforms for opacity-only alternatives without duplicating animation logic.
- **Anti-pattern**: Removing animation entirely when the user prefers reduced motion — the docs show keeping opacity feedback while dropping x/y movement.
- **Code/prompt excerpt**:
```javascript
import { useReducedMotion } from "motion/react"

export function Sidebar({ isOpen }) {
  const shouldReduceMotion = useReducedMotion()
  const closedX = shouldReduceMotion ? 0 : "-100%"

  return (
    <motion.div animate={{
      opacity: isOpen ? 1 : 0,
      x: isOpen ? 0 : closedX
    }} />
  )
}
```
- **Why it matters for THIS skill**: This is the exact React primitive the skill should export as its reduced-motion adapter — every tier's component wires through this before deciding whether to run a spatial or opacity-only transition.

### Source 2: GSAP Accessibility Docs — matchMedia dual-branch pattern
- **URL**: https://gsap.com/resources/getting-started/accessibility/
- **Pattern**: `gsap.matchMedia()` splits animation into two named branches — `no-preference` gets the full motion sequence; `reduce` gets a stripped opacity-only fallback — keeping both paths live and testable.
- **Anti-pattern**: Flashing animations with fast, hard cuts between colors; large swiping movements on x/y axes (vestibular triggers that should be removed at all tiers, not just reduced-motion mode).
- **Code/prompt excerpt**:
```javascript
let mm = gsap.matchMedia()
mm.add("(prefers-reduced-motion: no-preference)", (context) => {
  gsap.from(".box", {
     opacity: 0,
     rotation: 360,
     ease: "back.out"
  })
});
mm.add("(prefers-reduced-motion: reduce)", (context) => {
  gsap.from(".box", { opacity: 0})
});
```
- **Why it matters for THIS skill**: The dual-branch model is the right architecture for Tier 2–3 components — one code path per motion preference, not a single path with conditionals scattered inside it.

### Source 3: WCAG 2.3.3 — Animation from Interactions (SC 2.3.3, Level AAA)
- **URL**: https://www.w3.org/WAI/WCAG22/Understanding/animation-from-interactions.html
- **Pattern**: Any motion triggered by user interaction must be disableable unless the animation is essential to the information or functionality — this is the legal floor every tier must clear.
- **Anti-pattern**: Animations that are triggered by interaction, non-essential to function or information, and cannot be disabled by the user or their OS settings.
- **Code/prompt excerpt**:
```
"Motion animation triggered by interaction can be disabled, unless the
animation is essential to the functionality or the information being conveyed."

Sufficient techniques:
1. Using the CSS prefers-reduced-motion query to prevent motion (C39)
2. Using the CSS prefers-reduced-motion query in JavaScript to prevent motion (SCR40)
3. Allowing users to set a preference that prevents animation (Gx)
```
- **Why it matters for THIS skill**: The grader this skill exports must check that every tier's animations are either essential-to-function or have a disable path — this is the criterion that defines "pass" vs "fail" for the compliance layer.

### Source 4: WebKit Blog — Responsive Design for Motion (Six Vestibular Trigger Categories)
- **URL**: https://webkit.org/blog/7551/responsive-design-for-motion/
- **Pattern**: Six animation categories are known vestibular triggers (scaling/zooming, spinning, multi-speed parallax, plane shifting, peripheral movement, animated blur) — a site-type grader should flag these regardless of tier, and offer opacity substitutes.
- **Anti-pattern**: Removing ALL animation when reduced motion is set — the article explicitly says "only remove the animations you *know* to be vestibular triggers" and suggests serving a simpler alternative instead.
- **Code/prompt excerpt**:
```javascript
var motionQuery = matchMedia('(prefers-reduced-motion)');
function handleReduceMotionChanged() {
  if (motionQuery.matches) {
    /* adjust motion of 'transition' or 'animation' properties */
  } else { 
    /* standard motion */
  }
}
motionQuery.addListener(handleReduceMotionChanged);
handleReduceMotionChanged(); // trigger once on load if needed
```
Six trigger categories (exact from article):
```
1. Scaling and Zooming
2. Spinning and Vortex Effects
3. Multi-Speed or Multi-Directional Movement (parallax)
4. Dimensionality or Plane Shifting
5. Peripheral Motion
6. Animated Blurring
```
- **Why it matters for THIS skill**: Gives the grader a concrete checklist of pattern types that must be gated behind `prefers-reduced-motion: no-preference` regardless of what tier the site is on — a local business site on Tier 1 that includes parallax still fails this check.

### Source 5: Smashing Magazine — Designing for Reduced Motion Sensitivities (Animation Classification Table)
- **URL**: https://www.smashingmagazine.com/2020/09/design-reduced-motion-sensitivities/
- **Pattern**: Animation types split into three columns — triggering (remove), safe (keep), and substitution strategy (swap transform→opacity). The article also shows a concrete substitution: replace `bouncing` with `fade` rather than no animation.
- **Anti-pattern**: Keeping large movements, zooms, spinning, and parallax under reduced motion — these are explicitly marked as triggering and must be removed or replaced.
- **Code/prompt excerpt**:
```css
/* Default animation */
h2 {
  animation: bouncing 1.5s linear infinite alternate;
}

/* Reduced motion alternative — swap property, keep duration + easing */
@media (prefers-reduced-motion) {
  h2 {
    animation: fade 0.5s ease-in both;
  }
}
```
Classification table (verbatim):
```
Large movements, zooms, spinning  → ⚠️ Triggering  → Remove or significantly reduce
Parallax effects                  → ⚠️ Highly triggering → Remove entirely
Large page transitions            → ⚠️ Triggering  → Replace with crossfade
Color fades, opacity changes      → ✓ Safe         → Keep
Small scale changes               → ✓ Safe         → Keep
Hover effects                     → ✓ Safe         → Can remain in reduced mode
```
- **Why it matters for THIS skill**: This is the most directly usable classification table for building the grader's animation-type scoring — each pattern the skill generates can be tagged against this table to verify it has a valid reduced-motion path.

### Source 6: MDN — prefers-reduced-motion (Baseline spec + OS settings map)
- **URL**: https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion
- **Pattern**: The feature has two keyword values (`no-preference`, `reduce`); `@media (prefers-reduced-motion)` alone is equivalent to `reduce`. Baseline Widely Available since January 2020 across all major browsers.
- **Anti-pattern**: Animating layout-bound properties (height, width, background-color) under normal preference — the example explicitly swaps `pulse` (transform/scale) for `dissolve` (opacity) as the reduced-motion path, not just removing animation.
- **Code/prompt excerpt**:
```css
.animation {
  animation: pulse 1s linear infinite both;
  background-color: purple;
}

/* Tone down the animation to avoid vestibular motion triggers */
@media (prefers-reduced-motion: reduce) {
  .animation {
    animation: dissolve 4s linear infinite both;
    background-color: green;
  }
}

@keyframes pulse {
  0% { transform: scale(1); }
  25% { transform: scale(0.9); }
  50% { transform: scale(1); }
  75% { transform: scale(1.1); }
  100% { transform: scale(1); }
}

@keyframes dissolve {
  0% { opacity: 1; }
  50% { opacity: 0.3; }
  100% { opacity: 1; }
}
```
- **Why it matters for THIS skill**: Confirms the spec-level pattern the skill must export: not `animation: none` but a substituted opacity keyframe — this distinction drives the "substitute, don't strip" rule for every tier's reduced-motion path.

### Source 7: Chrome Developers — Scroll-Driven Animations (CSS animation-timeline)
- **URL**: https://developer.chrome.com/articles/scroll-driven-animations/
- **Pattern**: CSS `animation-timeline: scroll()` and `animation-timeline: view()` deliver scroll-progress and element-visibility animations with zero JavaScript, making them viable for Tier 3 scroll reveals on content-heavy sites without a JS animation library dependency.
- **Anti-pattern**: Setting `animation-duration` in seconds when using a Scroll Progress Timeline — "you must set `animation-duration` to `auto`" since time-based duration has no meaning on a scroll axis.
- **Code/prompt excerpt**:
```css
/* Reading progress bar — zero JS */
@keyframes grow-progress {
  from { transform: scaleX(0); }
  to { transform: scaleX(1); }
}

#progress {
  position: fixed;
  left: 0; top: 0;
  width: 100%; height: 1em;
  background: red;
  transform-origin: 0 50%;
  animation: grow-progress auto linear;
  animation-timeline: scroll();
}

/* Reveal on entry — zero JS */
@keyframes reveal {
  from { opacity: 0; clip-path: inset(0% 60% 0% 50%); }
  to { opacity: 1; clip-path: inset(0% 0% 0% 0%); }
}

.revealing-image {
  animation: auto linear reveal both;
  animation-timeline: view();
  animation-range: entry 25% cover 50%;
}
```
- **Why it matters for THIS skill**: Tier 3 scroll reveals on documentation, blog, and content sites should default to the CSS scroll-driven path (no runtime cost, no GSAP license needed) — this source provides the exact patterns to ship as those components.

### Source 8: Chrome Developers — View Transitions API (SPA + MPA patterns)
- **URL**: https://developer.chrome.com/docs/web-platform/view-transitions/same-document
- **Pattern**: `document.startViewTransition()` wraps any DOM update with a cross-fade or custom CSS animation; directional transitions use named `types` to switch CSS keyframes based on navigation direction (forward/back), making route changes feel native without a JS animation library.
- **Anti-pattern**: Starting a view transition when `e.hasUAVisualTransition` is already true — on mobile browsers that provide native transitions, triggering a JS transition doubles the animation and jars the user.
- **Code/prompt excerpt**:
```javascript
// Basic SPA route transition
function handleClick(e) {
  if (!document.startViewTransition) {
    updateTheDOMSomehow();
    return;
  }
  document.startViewTransition(() => updateTheDOMSomehow());
}

// Directional transitions (forward/back)
const direction = determineBackwardsOrForwards();
const t = document.startViewTransition({
  update: updateTheDOMSomehow,
  types: ['slide', direction],
});

// CSS for directional types
html:active-view-transition-type(forwards) {
  &::view-transition-old(content) { animation-name: slide-out-to-left; }
  &::view-transition-new(content) { animation-name: slide-in-from-right; }
}
```
- **Why it matters for THIS skill**: Tier 3 "signature route transitions" on marketing sites and portfolio sites should use the View Transitions API as the default implementation — it's the zero-library path for the moves that previously required GSAP page transition plugins.

## Key Synthesis for Skill Rebuild

**What the sources confirm:**

1. **Floor rule (non-negotiable, all tiers, all site types):** Any animation triggered by interaction must have a disable path via `prefers-reduced-motion` or a user preference. This is WCAG 2.3.3. The grader fails any component that lacks this.

2. **Substitute, don't strip:** The consistent pattern across WebKit, GSAP, Motion.dev, and Smashing is to swap the animating *property* (transform → opacity) while keeping duration and easing. Stripping to `animation: none` is the wrong default.

3. **Six trigger types that break the floor regardless of tier:** scaling/zooming, spinning, multi-direction parallax, plane shifting, peripheral motion, animated blur. These must be gated behind `no-preference` even on Tier 1 "minimal" sites. A local business site with a parallax hero fails the grader.

4. **Zero-JS paths exist for Tier 3:** CSS `animation-timeline: view()` handles scroll reveals; `@view-transition { navigation: auto }` handles MPA route transitions. The skill should default to these before reaching for GSAP or Framer Motion — they have lower runtime cost and no library dependency.

5. **Site-type signal from source gaps:** Material, Carbon, and Fluent all failed to render content, which itself is evidence — their motion systems are design-tool-centric (Figma tokens, not CSS exports). The skill's site-type taxonomy should be sourced from the actual usage patterns (scroll-driven for content, micro-interactions for dashboard, signature transitions for portfolio/marketing) rather than mapped onto design-system surface categories.
