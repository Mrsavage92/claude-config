## What the skill is supposed to produce

Concrete `Edit`/`Write` tool calls that add purposeful entrance, micro-interaction, state-transition, and feedback animations to a specified target file or feature — using `motion/react` primitives (or GSAP for Tier 3) and following the host project's site-type tier ceiling from `/web-animations` site-profile.json.

NOT: prose advice telling the user what they could add. The skill must produce code changes.

## Success criteria (each verifiable from the artifact alone)

1. **Concrete edits, not advice.** Artifact contains at least 3 specific `Edit`/`Write` tool calls (or proposed file-level diffs) against the target file. Recommendations expressed only as bullet-list prose = FAIL.
2. **Respects project tier ceiling.** If the target route is at SaaS-App MAX 2, the artifact MUST NOT propose Tier 3 moves (PinnedSection, Lenis, GSAP scroll-driven, cursor parallax). Any Tier 3 proposal on a Tier 2-capped route = FAIL.
3. **Reduced-motion guard on every Tier 2+ proposal.** Each spring / scale / large-y proposal must include `useReducedMotion()` or a `@media (prefers-reduced-motion: reduce)` fallback in the proposed code.
4. **Uses motion/react, not framer-motion.** Imports proposed by the artifact must read `from 'motion/react'`. Any `from 'framer-motion'` = FAIL.
5. **One hero / signature moment, not maximalism.** Skill's own SKILL.md says "One well-orchestrated experience beats scattered animations everywhere." Artifact must apply that — designate ONE hero moment, then a tight set of micro-interactions, not 15 animations on one page.
6. **No banned phrases** in the artifact (`comprehensive, robust, production-ready, world-class, premium, perfect, best-in-class, enterprise-grade, battle-tested, seamless, cutting-edge`).
7. **Performance budget acknowledged.** Artifact mentions or shows compositor-only properties (transform, opacity, clip-path) only. No animated `width`/`height`/`top`/`left`/`margin`/`padding`.
8. **A11y beyond reduced-motion**: focus management on modal/route, screen-reader-aware micro-interactions where applicable (no aria-live spam on decorative animation).

## Constraints

- **Target host project**: AuditHQ (`C:/Users/Adam/audit-genius/`)
- **Stack**: Vite + React 18 + React Router v6 + `motion` v12 (`motion/react`) + Tailwind. NOT Next.js. No `'use client'` needed (harmless if present).
- **Site profile**: hybrid (see `~/audit-genius/.evolution/site-profile.json`). Per-route ceilings already declared. The skill MUST read this file.
- **Tooling available**: motion/react, tailwind motion-reduce: variants. gsap + lenis NOT installed by default — skill should not propose them on app routes.
- **Dependencies unavailable**: framer-motion legacy (renamed to `motion`); R3F/Three (not in deps).
- **Must-haves locked**: every Tier 2+ proposal needs reduce-motion guard. Above-the-fold uses `animate="visible"` (mount), not `whileInView`. `viewport={{ once: true }}` on all scroll reveals.
- **Banned approaches**: custom cursors by default (per `feedback_no_custom_cursor_by_default`), Bruno Simon / Awwwards / SOTM defaults on non-portfolio sites (per `feedback_taste_calibration`).

## Edge cases to cover (artifact must demonstrate)

- Target is a SaaS-app route (`/audit/new`, `/pricing`, `/settings`) — proposals stay at Tier 2 max.
- Target is a marketing route (`/`, `/customers`) — Tier 3 selectively justified for one signature moment.
- Target page already has heavy motion (e.g. `Index.tsx` aurora + scroll-driven) — skill should ADD micro-interactions where missing, not duplicate the hero treatment.
- Above-the-fold elements use mount entrance, not scroll-triggered.
- Form submit / loading states get spinner + state-transition animation.
- Button feedback (hover / tap / loading) uses spring physics with snappy preset.
