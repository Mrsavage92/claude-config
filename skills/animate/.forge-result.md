## What the rebuild closed

1. **`Skill('impeccable')` literal invocation** — Cardinal Rule 3 now explicitly says "LITERAL invocation, not Read on its SKILL.md". The rebuilt skill DEMANDS this when invoked normally. The forge artifact agents can't actually call other skills from inside their own context (harness limitation) — this is structural, not a skill defect. Real-world invocations of `/animate` (where the user runs the skill in their main thread) WILL use the Skill tool.
2. **`background-color` transitions banned by default** — Cardinal Rule 8 added after iter 2 caught a Floor-5 violation in the artifact. Now even small-surface `transition-colors` is forbidden; replacement must use overlay-opacity or filter-brightness.
3. **ARIA-state replacement requirement** — Cardinal Rule 5 + new `references/aria-state-replacements.md`. Every motion-removal proposal that strips a state indicator must explicitly add the ARIA equivalent (`aria-pressed`, `aria-expanded`, `aria-current`, `role=switch`).
4. **Allowed-moves audit (Phase 2.5)** — added after iter 2 caught SpringButton silently omitted. Artifact must list every Allowed move from the recipe row and mark each PROPOSED or SKIPPED with rationale.
5. **Per-role recipes** — new `references/per-role-recipes.md` with 20 route role × allowed/banned/hero combinations. Drives ALL proposals.
6. **Floor citation on numeric thresholds** — Cardinal Rule 9 (in the rebuild). Any scale/duration/y-offset value cites the floor-rules.md line that authorises it.
7. **Tier ceiling read from `.evolution/site-profile.json`** — hard cap before any proposal.
8. **`motion/react` not `framer-motion`** — Cardinal Rule 4. Closed.

## What still gates a PASS (2-point gap remaining)

1. **Focus-visible:ring not re-confirmed after class swap** (−2). When the rebuilt skill produces a removal proposal that replaces a state-bearing motion with a static class-toggle, the artifact should explicitly confirm that focus-visible styling persists. This is a NEW rule the skill doesn't yet have. Adding it would push the next iter to ~90.
2. **Cascade removal as discrete proposal** (−3). When a removal proposal orphans a hook or import, the cleanup must be a numbered proposal, not a verbal note. The skill has the rule ("Removal proposals must include a cascade check") but doesn't currently REQUIRE it to be a discrete proposal. Iter 3 documented the orphan verbally — the scoring agent wanted a separate Proposal N+1.

Both are easy patches but skill-forge's 3-rebuild cap halts further iteration.
