### Source 1: Motion (Framer Motion v12) — useReducedMotion Hook
- **URL**: https://motion.dev/docs/react-use-reduced-motion
- **Pattern**: Call `useReducedMotion()` at the component root to get a boolean, then branch `animate` prop values inline — keeping a single render path rather than two separate component trees, and substituting opacity-only changes for positional transforms under reduced motion.
- **Anti-pattern**: Maintaining two separate render branches (one with motion, one without) couples animation logic to component structure and makes future changes harder.
- **Code/prompt excerpt**:
```jsx
export function Sidebar({ isOpen }) {
  const shouldReduceMotion = useReducedMotion()
  const closedX = shouldReduceMotion ? 0 : "-100%"

  return (
    <motion.div animate={{
      opacity: isOpen ? 1 : 0,
      x: isOpen ? 0 : closedX
    }} />
  )
}
```
- **Why it matters for THIS skill**: Any code changes the skill proposes must include a `useReducedMotion` branch; without it the enhancement fails the accessibility floor.

### Source 2: Motion (Framer Motion v12) — Variants & Entrance Animations
- **URL**: https://motion.dev/docs/react-animation
- **Pattern**: Define a `variants` object with named states (`hidden`/`visible`), pass `initial`/`animate` as string labels, and use `AnimatePresence` wrapping `exit` props for removal — variants propagate down the component tree so parent orchestration controls stagger timing.
- **Anti-pattern**: Inlining identical animation values on every list child instead of using variants + `staggerChildren` on the parent.
- **Code/prompt excerpt**:
```jsx
const list = {
  visible: {
    opacity: 1,
    transition: {
      when: "beforeChildren",
      delayChildren: stagger(0.3),
    },
  },
}

<AnimatePresence>
  {isVisible && (
    <motion.div
      initial={{ opacity: 0 }}
      animate={{ opacity: 1 }}
      exit={{ opacity: 0 }}
    />
  )}
</AnimatePresence>
```
- **Why it matters for THIS skill**: The skill's entrance and state-transition proposals need the variants+stagger pattern, not per-element inline values.

### Source 3: Motion (Framer Motion v12) — Gesture Micro-interactions
- **URL**: https://motion.dev/docs/react-gestures
- **Pattern**: Attach `whileHover` and `whileTap` directly to interactive elements with small scale transforms (hover up, tap down), then use variant propagation so nested SVG paths animate in sync without separate event handlers.
- **Anti-pattern**: Using `onMouseEnter`/`onMouseLeave` with `useState` to drive animation values when `whileHover` achieves the same with less code and no re-renders.
- **Code/prompt excerpt**:
```jsx
<motion.button
  whileTap="tap"
  whileHover="hover"
  variants={buttonVariants}
>
  <svg>
    <motion.path variants={iconVariants} />
  </svg>
</motion.button>

// standalone examples:
<motion.a whileHover={{ scale: 1.2 }} />
<motion.button whileTap={{ scale: 0.9, rotate: 3 }} />
```
- **Why it matters for THIS skill**: Micro-interaction proposals (button press, card hover) should use `whileHover`/`whileTap` with variant propagation, not state-driven style objects.

### Source 4: Motion (Framer Motion v12) — Layout Animations
- **URL**: https://motion.dev/docs/react-layout-animations
- **Pattern**: Add `layout` prop to any `motion.*` element to automatically animate its position/size when it changes between renders; use `layoutId` to create shared-element transitions across components without manually tracking coordinates.
- **Anti-pattern**: Manually animating `width`, `height`, `top`, or `left` properties to achieve layout transitions — these trigger reflow and produce janky results.
- **Code/prompt excerpt**:
```jsx
// State transition — layout animates justify-content shift
<motion.div
  layout
  style={{ justifyContent: isOn ? "flex-start" : "flex-end" }}
/>

// Shared element transition across components
<motion.li layoutId="item" />

// Modal with AnimatePresence
<AnimatePresence>
  {isOpen && <motion.div layoutId="modal" />}
</AnimatePresence>

// Custom timing
<motion.div layout transition={{ duration: 0.3 }} />
```
- **Why it matters for THIS skill**: State-transition proposals (tab switches, expanding cards) should reach for `layout`/`layoutId` before any manual coordinate tracking.

### Source 5: MagicUI — BlurFade Component (verbatim source)
- **URL**: https://github.com/magicuidesign/magicui (registry/magicui/blur-fade.tsx via public registry JSON)
- **Pattern**: Combine `useInView` with a `variants` object that includes a `filter: blur()` hidden state, and a `direction` prop that dynamically picks the x or y offset axis — a single reusable wrapper handles all four directional entrance variants plus an `inViewMargin` threshold for scroll-triggered firing.
- **Anti-pattern**: Hardcoding directional offsets and blur values inline per call site rather than parameterising them in a single reusable `BlurFade` wrapper.
- **Code/prompt excerpt**:
```tsx
const defaultVariants: Variants = {
  hidden: {
    [direction === "left" || direction === "right" ? "x" : "y"]:
      direction === "right" || direction === "down" ? -offset : offset,
    opacity: 0,
    filter: `blur(${blur})`,
  },
  visible: {
    [direction === "left" || direction === "right" ? "x" : "y"]: 0,
    opacity: 1,
    filter: `blur(0px)`,
  },
}
// ...
<motion.div
  ref={ref}
  initial="hidden"
  animate={isInView ? "visible" : "hidden"}
  exit="hidden"
  variants={combinedVariants}
  transition={{ delay: 0.04 + delay, duration, ease: "easeOut" }}
>
  {children}
</motion.div>
```
- **Why it matters for THIS skill**: This is the canonical scroll-triggered entrance wrapper pattern — the skill should propose a similar parameterised component rather than scattered per-element `whileInView` props.

### Source 6: Production Pricing Page — Framer Motion Variants (GitHub: beta-zapdev)
- **URL**: https://github.com/Jackson57279/beta-zapdev/blob/2903bb8eab9ebc441a3fc10629807670816b138f/components/pricing.tsx
- **Pattern**: A `containerVariants` parent with `staggerChildren: 0.1` drives `itemVariants` (`opacity: 0, y: 20` → `opacity: 1, y: 0`) on each pricing card; `whileHover` on each card uses `scale: 1.02` with a `boxShadow` change to signal interactivity without large viewport movement.
- **Anti-pattern**: Animating cards with `scale: 1.1` or large y-translations on hover — this shifts surrounding layout and reads as disruptive rather than inviting.
- **Code/prompt excerpt**:
```tsx
const containerVariants = {
  hidden: { opacity: 0 },
  visible: {
    opacity: 1,
    transition: { staggerChildren: 0.1 }
  }
}
const itemVariants = {
  hidden: { opacity: 0, y: 20 },
  visible: { opacity: 1, y: 0 }
}
// ...
<motion.div
  key={plan.tier}
  variants={itemVariants}
  whileHover={{
    scale: 1.02,
    boxShadow: "0px 0px 20px rgba(108, 82, 160, 0.3)",
  }}
>
```
- **Why it matters for THIS skill**: Anchors the "right amount of motion" calibration — scale 1.02 + shadow is the floor for SaaS card hover; scale 1.1 is the ceiling to avoid.

### Source 7: Smashing Magazine — Respecting Motion Preferences
- **URL**: https://www.smashingmagazine.com/2021/10/respecting-users-motion-preferences/
- **Pattern**: Write motion inside `@media (prefers-reduced-motion: no-preference)` blocks (opt-in) rather than stripping with a wildcard `* { animation: none !important }` (opt-out) — preserves subtle, usability-relevant transitions for reduced-motion users.
- **Anti-pattern**: The wildcard `* { animation: none !important; transition: none !important; }` rule — it silently overrides any per-element reduced-motion fallback lower in the cascade, removing even subtle colour/opacity feedback.
- **Code/prompt excerpt**:
```css
/* WRONG — wildcard blocks per-element fallbacks */
@media screen and (prefers-reduced-motion: reduce) {
  * {
    animation: none !important;
    transition: none !important;
    scroll-behavior: auto !important;
  }
}

/* RIGHT — opt-in, leaves fallbacks intact */
@media (prefers-reduced-motion: no-preference) {
  .some-element {
    animation: bounce 1200ms;
  }
}
```
```js
// Conditional library loading
const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)')
const loadGSAPAndInitAnimations = () => {
  if (prefersReducedMotion.matches) return
  import('gsap').then((object) => { /* init */ })
}
```
- **Why it matters for THIS skill**: The skill must emit opt-in motion blocks, never opt-out wildcards — this is the CSS-side mirror of the `useReducedMotion` JS pattern.

### Source 8: Smashing Magazine — Design for Reduced-Motion Sensitivities
- **URL**: https://www.smashingmagazine.com/2020/09/design-reduced-motion-sensitivities/
- **Pattern**: When removing a triggering transform, swap the animating CSS property (e.g. `translateY` → `opacity`) while preserving the duration and easing values — this keeps the motion purposeful and design-consistent rather than eliminating all feedback.
- **Anti-pattern**: Large parallax effects and spinning/zooming viewport-dominating transitions — every person with motion sensitivities interviewed flagged these as universally triggering, regardless of `prefers-reduced-motion` setting.
- **Code/prompt excerpt**:
```css
/* Problematic — large vertical movements */
h2 {
  animation: bouncing 1.5s linear infinite alternate;
}
.plant1 {
  transform: translateY(-100%);
  transition: 0.6s ease-both;
}

/* Safe alternative — swap property, keep timing */
@media (prefers-reduced-motion: reduce) {
  h2 {
    animation: fade 0.5s ease-in both;
  }
  .plant1 {
    transform: translateX(0);
    opacity: 0;
  }
}
```
- **Why it matters for THIS skill**: The skill should never propose `translateY` or viewport-scale animations without a `reduced` fallback that swaps to opacity only.

### Source 9: Josh Comeau — CSS Transitions
- **URL**: https://www.joshwcomeau.com/animation/css-transitions/
- **Pattern**: Asymmetric transition timing (slow exit, fast entry) maps to interaction semantics — hover-in at 125ms feels responsive, hover-out at 450ms feels deliberate; `transition-delay` on the exit but not the enter prevents accidental trigger while mousing across.
- **Anti-pattern**: `transition: all` — explicitly called out as something "I recommend not using" because future property additions animate unexpectedly; also avoid animating `width`/`height` when `transform` achieves the same visual result at compositor cost.
- **Code/prompt excerpt**:
```css
/* Action-driven asymmetric timing */
.btn {
  transition: transform 450ms;
}
.btn:hover {
  transition: transform 125ms;
  transform: translateY(-10px);
}

/* Dropdown delay — prevent accidental trigger */
.dropdown {
  opacity: 0;
  transition: opacity 400ms;
  transition-delay: 300ms;
}
.dropdown-wrapper:hover .dropdown {
  opacity: 1;
  transition: opacity 100ms;
  transition-delay: 0ms;
}
```
- **Why it matters for THIS skill**: Micro-interaction proposals must specify asymmetric durations matching interaction direction, not a single `transition` value for both enter and exit.
